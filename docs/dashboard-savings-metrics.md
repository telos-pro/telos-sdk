# Savings Dashboard Metrics

Entry point: `GET /__telos/dashboard` (embedded in the proxy) or the CLI:

```sh
telos dashboard --usage-log ~/.telos/usage.jsonl --out savings.html
```

Source: [scripts/build_savings_dashboard.py](../scripts/build_savings_dashboard.py).

> **Dashboard perspective**: aimed at the "user/boss" — it answers "after adopting TELOS, how many tokens were saved, how much money was saved".
> The data source is a jsonl-format `usage_log`, one line per call with the `normalized` + `raw_usage` fields.

---

## 1. Billing model: converting tokens to dollars

### 1.1 The four token buckets (from Anthropic's `usage` field, already normalized)

| Bucket | Source | Meaning |
|---|---|---|
| `raw_input` | `usage.input_tokens` | prompt tokens that **missed the cache and did not write the cache** |
| `cache_read` | `usage.cache_read_input_tokens` | prompt tokens that **hit the cache** (the key to savings) |
| `cache_write` | `usage.cache_creation_input_tokens` | prompt tokens newly written to the cache this call (for reuse next time) |
| `output` | `usage.output_tokens` | tokens generated by the model |

> Key constraint: Anthropic's `input_tokens` field is **only `raw_input`**, and does not include `cache_read`
> or `cache_write`. So
> **total prompt tokens = raw_input + cache_read + cache_write**.
> Every `hit%` denominator on the dashboard is this sum (the old implementation used only raw_input as the denominator, which overestimates the hit rate).

### 1.2 Price table (USD per 1M tokens, 2026 public prices)

Source: [scripts/build_savings_dashboard.py:56-76](../scripts/build_savings_dashboard.py#L56-L76).

| Model prefix | input | cache_read | cache_write 5m | cache_write 1h | output |
|---|---:|---:|---:|---:|---:|
| `claude-opus-4-7` / `4-6` | 5.00 | 0.50 | 6.25 | 10.00 | 25.00 |
| `claude-opus-4-5` / `4`   | 15.00 | 1.50 | 18.75 | 30.00 | 75.00 |
| `claude-sonnet-4-6` / `4-5` / `4` | 3.00 | 0.30 | 3.75 | 6.00 | 15.00 |
| `claude-haiku-4-5` / `4`  | 1.00 | 0.10 | 1.25 | 2.00 | 5.00 |
| `gpt-5` / `gpt-5.1`       | 5.00 | 1.25 | 0 | 0 | 15.00 |
| `deepseek-chat` / `v3`    | 0.27 | 0.07 | 0 | 0 | 1.10 |
| `_default` (unrecognized) | 3.00 | 0.30 | 3.75 | 6.00 | 15.00 |

**Anthropic's cache billing rules** (already encoded in the price table):

```
cache_read price = 0.10 × input price        cache hit → 90% off
cache_write 5m   = 1.25 × input price         short-TTL write → pay 25% more
cache_write 1h   = 2.00 × input price         long-TTL write → pay 100% more
```

The 5m / 1h split of `cache_write` comes from `raw_usage.cache_creation.ephemeral_{5m,1h}_input_tokens`.
If missing, **everything is counted as 5m** (conservatively underestimating the 1h portion).

### 1.3 The actual cost of a single call

[`_cost_usd`](../scripts/build_savings_dashboard.py#L116-L129):

```
cost = raw_input  × input_price
     + cache_read × cache_read_price
     + cache_write_5m × cache_write_5m_price
     + cache_write_1h × cache_write_1h_price
     + output     × output_price
```

### 1.4 Counterfactual cost ("if TELOS were off")

[`_counterfactual_cost_usd`](../scripts/build_savings_dashboard.py#L132-L143):
assume the prompt content stays the same but **all `cache_control` is removed** — all prompt tokens are billed at the
base `input` price, with no cache_read discount and no cache_write premium:

```
counterfactual = (raw_input + cache_read + cache_write) × input_price
               + output × output_price
```

### 1.5 Amount saved

[`_saved_usd_for_call`](../scripts/build_savings_dashboard.py#L146-L164):

```
saved = counterfactual − actual
      = cache_read     × (input_price − cache_read_price)        ← earned back by cache hits
      + cache_write_5m × (input_price − cache_write_5m_price)    ← 5m write premium, negative
      + cache_write_1h × (input_price − cache_write_1h_price)    ← 1h write premium, more negative
```

**Important**: for Anthropic, the `cache_write` term is a **negative contribution** (a write costs 25–100% more than the base price).
Only when the `cache_read` volume is large enough (i.e. the cache is reused many times) does the total turn positive. This is the key correction relative to earlier
implementations — the old version only counted the cache_read discount, overestimating the money saved.

## 2. Top Hero ("with TELOS, actual" perspective)

| Hero | Calculation |
|---|---|
| **tokens saved (cache hits)** | `total.cache_read` (total prompt tokens that hit the cache) |
| **cost saved (estimated)** | `total.saved_usd` (the accumulation in §1.5 above) |

Subtitles:
- `cache hits are X% of total prompt tokens` — `cache_read / (raw_input + cache_read + cache_write)`
- `with TELOS off you'd expect to pay $A · actually paid $B · saved X%` — counterfactual vs actual vs ratio

When switched to the "without TELOS · counterfactual" perspective, the hero becomes:
- **prompt tokens (no cache)** = `raw_input + cache_read + cache_write` (all at the input price)
- **cost (no TELOS)** = the counterfactual cost from §1.4

## 6. Breakdown by harness / model / session

The three tables have the same structure, sorted by `cache_read` descending for the top N rows (harness/model take the top 12 rows, session the top 15).

| Column | Meaning |
|---|---|
| key | `harness` / `model` / `session_id` |
| `calls` | the number of calls under this key |
| `raw_input` | this key's cumulative `raw_input` |
| `cache_read` | this key's cumulative `cache_read` (green, the dashboard's core column) |
| `cache_write` | this key's cumulative `cache_write` |
| `hit%` | `cache_read / (raw_input + cache_read + cache_write)` (note the denominator includes cache_write, unlike earlier implementations) |
| `saved $` | a bar + a number. The bar takes the "largest saved amount" in this table as 100% |

> The session table keeps only the top 15 rows — long sessions accumulate high cache_read and new sessions rank lower;
> to see the full set use `/__telos/developer.json`.

## 7. Perspective switching

Three buttons at the top of the page:

| Mode | What you see |
|---|---|
| **actual (TELOS on)** | the real four-color token mix, hit rate, actual saved $ |
| **counterfactual (TELOS off)** | treats all prompt tokens as raw_input, estimating total cost at the input price |
| **side by side** | the counterfactual big number on the left, the actual big number on the right, with a line at the bottom "net saved $X · X% off" |

The choice is persisted via `localStorage.telos.dashboard.mode`.

## 8. Data source schema

`usage_log` is one JSON record per line ([docs/User-guide.md §7.1](User-guide.md)). The fields the dashboard uses:

| Field | Required | Description |
|---|---|---|
| `normalized.raw_input` | ✅ | int |
| `normalized.cache_read` | ✅ | int |
| `normalized.cache_write` | ✅ | int |
| `normalized.output` | ✅ | int |
| `raw_usage.cache_creation.ephemeral_5m_input_tokens` | no | falls back to the 5m price if missing |
| `raw_usage.cache_creation.ephemeral_1h_input_tokens` | no | 1h not counted if missing |
| `model` | recommended | used to look up the price table; unrecognized falls back to `_default` (the Sonnet price tier) |
| `harness` | recommended | used for by_harness grouping |
| `session_id` | recommended | used for unique-sessions and by_session grouping |
| `ts` | recommended | a float unix timestamp, determines the timeline bucket |

## 9. Common interpretation questions

**Q: Does more cache_read mean more savings?**
Yes. A token that hits the cache pays only 10% of the price; every extra 1M tokens of cache_read saves $4.50 on Opus 4.7.

**Q: Is more cache_write always better?**
**Not necessarily**. Every 1M tokens written to the 5m cache costs $6.25 (25% more than input's $5), and to the 1h cache costs $10
(100% more). **Only when this cache is subsequently hit many times** can the write premium be amortized by the hit discount.
The dashboard's `saved $` already counts this premium as a negative contribution.

**Q: Why is cache_write counted in the hit% denominator?**
Because Anthropic's `input_tokens` field refers only to the part that missed and did not write the cache,
the true "total prompt volume" = `raw_input + cache_read + cache_write`. Using a smaller denominator overestimates the hit rate.

**Q: Is the counterfactual price overestimated?**
The counterfactual = the case with all `cache_control` turned off, **containing no** cache discount or premium at all. This is the
cleanest comparison baseline. TELOS's value is shrinking the gap (counterfactual − actual).

**Q: How is an unrecognized model handled?**
It falls to `_default` (estimated at the Sonnet price tier). In this case `saved $` is only an "approximate estimate",
and should not be treated as a precise bill.
